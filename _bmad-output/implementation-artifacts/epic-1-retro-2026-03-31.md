## Successes & Strengths

### 1. Consistent Architecture from Day One
The monorepo scaffolding (Story 1.1) established patterns that every subsequent story followed: Turborepo pipelines, shared packages, snake_case/camelCase naming conventions. No story deviated from these foundations.

 The "boring technology" approach paid off — zero architectural rewrites needed across 7 stories.

### 2. Design System as a Force Multiplier
Story 1.4's Obsidian design system (Button, Card, StatusMessage, Radix wrappers) dramatically accelerated UI development in Stories 1.5-1.7. Each subsequent story reused components instead of building from scratch. The CVA variant system for Button was particularly effective — one component served 5 stories.

 The `reducedMotion` prop pattern established in 1.4 propagated cleanly through all animated components in subsequent stories.

 Test quality remediation (87/100 → 97/100) established BDD Given/When/Then naming + `[X.X-UNIT-XXX]` traceability IDs as the project standard.

 Every story after 1.4 adopted this pattern.

 Accessibility testing via `vitest-axe` became a standard gate. This was the single most impactful story in the epic — it created the "floor" that everything else stands on.

### 3. Security-First RLS Implementation
Story 1.3's RLS implementation was exceptionally thorough. The team discovered 10 key PostgreSQL + asyncpg limitations during implementation, documented each one, and built around them. The `test_rls_user` non-superuser role for RLS verification demonstrated deep security understanding. The `set_config()` transaction-scoping decision (using `true` not `false`) prevented context leakage in pooled connections. This was the hardest story in the epic and the team delivered it with the highest quality bar.

 Test quality score: 82/100 (Good).

### 4. Test Coverage Scale
Test count grew from ~20 tests (Story 1.1) to 640+ tests (Story 1.7). Each story added tests proportionally. Test quality reviews were run consistently (stories 1.3, 1.4, 1.5, 1.6, 1.7). The `LeadFactory` and `createTranscriptEntry()` data factory patterns emerged for centralized test data generation.

 The `CI_TIMEOUT_MULTIPLIER` env var for performance thresholds showed CI-awareness.

### 5. Code Review Culture
Every story from 1.2 onwards received adversarial code review (Blind Hunter, Edge Case Hunter, Acceptance Auditor). Story 1.7 had 14 patch findings — all fixed. The code review process caught real security issues (case-insensitive Bearer token, exact path matching to prevent `/webhooks-other` bypass), visual bugs (disabled opacity too low for dark theme), and architectural concerns (double commit in transaction). The review process matured over the epic.

 Early reviews (1.2) found basic issues. Later reviews (1.7) caught nuanced threshold boundary bugs and race conditions.

## Challenges & Growth Areas

### 1. SQLModel `table=True` Constructor Silently Ignores kwargs (Stories 1.3, 1.6, 1.7)
**Impact:** High — The same bug appeared in 3 separate stories. When constructing `TenantModel` subclasses with `table=True`, positional kwargs are silently ignored by SQLModel/Pydantic `__init__`. The fix was to use `model_validate()` with camelCase aliases. This was discovered in Story 1.3 and documented, but the pattern re-surfaced in Stories 1.6 and 1.7, suggesting the learning didn't fully propagate. Future stories should start with `model_validate()` as the default approach. Root cause: SQLModel is still in beta (v0.0.x) with known bugs around `table=True` model construction. The AliasGenerator adds another layer where camelCase keys must be used.

 
**Root Cause:** Under-documented behavior in a beta library combined with AliasGenerator mapping.
**Action:** Document `model_validate()` as the standard pattern in project-context.md. Add a linter or pre-commit check.

### 2. Corrupt Test Files (Stories 1.5, 1.6)
**Impact:** Medium — Two test files (`branding.test.ts`, `test_settings.py`) were generated in a corrupt state and had to be rewritten from scratch. The exact nature of corruption wasn't fully documented, but it appeared to be an LLM output truncation issue (truncated code, broken syntax). This cost time in both writing and debugging. 
**Root Cause:** Likely LLM output length limitations or generation artifacts.
**Action:** Consider adding a post-generation validation step that checks for syntax errors in generated test files before marking them complete.

### 3. Server Action Auth Pattern Inconsistency (Stories 1.2 vs 1.5+)
The early server actions in `client.ts` lacked `Authorization: Bearer` headers. The correct pattern was established in `branding.ts` (Story 1.5) using `auth().getToken()`. Every story after 1.5 had to explicitly call out "DO NOT follow the client.ts pattern." This inconsistency caused confusion and required repeated corrections in dev notes. 
**Root Cause:** Stories were written by different agents/sessions without a canonical reference implementation.
**Action:** The `branding.ts` pattern should be documented as the canonical auth pattern in project-context.md. Consider refactoring `client.ts` to match.

### 4. Barrel Export Gaps (Discovered in Story 1.4)
`packages/types/tenant.ts` was not re-exported from `packages/types/index.ts`. This was discovered during Story 1.4 and fixed in Story 1.5. The gap existed for multiple stories without anyone noticing. 
**Root Cause:** No automated check for barrel export completeness.
**Action:** Add a `turbo run types:sync` validation step that checks all `.ts` files in `packages/types/` are re-exported from `index.ts`.

### 5. Testing on Stories 1.5 and 1.6 — Combined Mistakes
As noted by the Project Lead (team mantis a), testing issues on stories 1.5 and 1.6 required extra time to resolve. The issues included corrupt test files, `"use client"` modules requiring `await import()` in Vitest, JWT orgs dict key mismatches (`org:123` vs `org_123`), and stale sessionStorage cache in BrandingProvider tests. These were implementation-specific issues rather than design flaws. 
**Root Cause:** Complex interactions between Next.js `"use client"` modules, Vitest test environment, and Clerk test fixtures.
**Action:** Establish a shared test utilities file for common Clerk/Vitest patterns to prevent re-discovery in each story.

## Key Insights

1. **Foundation stories take disproportionate effort** — Stories 1.1-1.3 (infrastructure) were harder per story than Stories 1.4-1.7 (features). The RLS story (1.3) was the hardest single story with 10 key discoveries about PostgreSQL + asyncpg limitations. Once the foundation was solid, feature stories moved faster. This is expected but worth noting for Epic 2 estimation.

2. **Code reviews pay for themselves** — The adversarial code review process consistently caught real bugs: security issues (case-insensitive Bearer, path bypass), data integrity (double commit, stale cache), and UX issues (disabled opacity on dark theme). Every review identified actionable findings that were fixed before merge.

3. **Test quality scores are consistent at 82/100** — Multiple stories scored exactly 82/100 on initial test quality review, suggesting the review rubric has a consistent threshold. After BDD remediation, scores rose to 97/100. The gap is consistently the same: missing BDD naming, missing traceability IDs, missing data factories, and real async handling (fake timers vs real setTimeout).

4. **E2E tests remain fragile** — Multiple E2E test files are marked as "skip" or "pending" because they require Clerk test fixtures. This is a known gap across the project. Clerk's `@clerk/testing` requires specific environment setup that hasn't been automated yet.

5. **Technical debt is tracked but not alarming** — The 5 acknowledged debt items (race condition, VARCHAR vs JSONB, base64 logos, calendar billing, webhook auth) are documented and prioritized. None are blocking for Epic 2, though the webhook auth will need resolution before Story 2.1.

## Significant Discoveries

### No Fundamental Plan Changes Required
Epic 1 did not reveal any architectural assumptions that were wrong. The monorepo structure, Clerk hierarchy, RLS-based tenancy, and Obsidian design system all validated correctly. The plan for Epic 2 remains sound. However, two technical discoveries affect Epic 2 directly:

1. **Vapi webhook auth** — Story 1.7 noted that Epic 2 will need server-to-server webhook authentication without Clerk JWT. This needs design before Story 2.1.
2. **asyncpg limitations** — The `set_config()` pattern and single-statement-per-text() constraint will affect how Epic 2's real-time pipelines interact with PostgreSQL.
