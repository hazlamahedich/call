## Successes & Strengths

### 1. TTSOrchestrator Architecture: Provider Abstraction Done Right

The TTS provider abstraction (Story 2.3) is the crown jewel of Epic 2. The `TTSOrchestrator` class delivers:
- **Abstract interface** for ElevenLabs and Cartesia with identical API surfaces
- **Latency-based health tracking** with automatic fallback when P95 exceeds thresholds
- **Circuit breaker** with configurable cooldown to prevent oscillation
- **Session state management** with TTL-based cleanup to prevent memory leaks
- **Score: 97/100** on code review — the highest in the project

> The strict abstract interface approach paid off. Provider-specific API quirks (ElevenLabs streaming vs. Cartesia chunked) were hidden behind a clean `synthesize()` contract. No downstream code needed to know which provider was active.

### 2. Adversarial Code Review Process Matured

Epic 2 continued the 3-layer adversarial review (Blind Hunter, Edge Case Hunter, Acceptance Auditor) with measurably more sophisticated catches:
- **Story 2.4:** Caught tenant race condition vulnerability (P0 security), 10x load test discrepancy, silent data loss in telemetry queue
- **Story 2.5:** Found 17 issues across 4 severity levels — 3 critical, including component-can't-receive-voice-events integration gap and memory leak from timer race conditions
- **Story 2.6:** Identified story pivot opportunity based on industry analysis (calibration → presets)

> Reviews evolved from catching syntax bugs (Epic 1) to catching architectural blind spots and market alignment issues. The review process is now a strategic tool, not just a quality gate.

### 3. Story 2.6 Pivot: Market-Informed Decision Making

Story 2.6 was originally "Pre-Flight Calibration Dashboard" with complex sliders. During adversarial review, the team discovered that industry leaders (Vapi, Retell, ElevenLabs) all use presets, not calibration. The pivot:
- Reduced onboarding time from 10 minutes to 2 minutes (80%)
- Reduced implementation complexity by 40% (fewer lines of code)
- Reduced user decisions from 6+ slider adjustments to 2 clicks

> This is the first time in the project that a story was redesigned based on competitive analysis rather than internal feedback. It validates the importance of looking outside the codebase.

### 4. Telemetry System Design: Non-Blocking by Architecture

Story 2.4's telemetry system achieved the <2ms push guarantee through architectural decisions, not optimization tricks:
- In-memory `asyncio.Queue` with fire-and-forget push
- Batch worker with 1-second collection deadlines
- Graceful degradation — events dropped rather than blocking voice pipeline
- Explicit documentation of the fire-and-forget tradeoff

> The team chose to accept occasional data loss over voice pipeline latency. This tradeoff was documented, tested (AC 8.5 degradation visibility), and the monitoring dashboard ensures losses are detectable.

### 5. Frontend Visual Quality: Pulse-Maker Component

Story 2.5 demonstrated mature frontend practices:
- Pure CSS animations (60fps, GPU-accelerated) — no JavaScript animation overhead
- Binary volume state with exponential decay for natural-feeling transitions
- `prefers-reduced-motion` support (WCAG AAA) baked in from the start
- Screen reader support with `role="status"` and visually-hidden state text
- 17-issue adversarial review resulted in zero regressions

> The component was designed as a pure visual layer with no backend coupling — it subscribes to transcript entries and reacts. This separation of concerns made it trivially testable and independently deployable.

## Challenges & Growth Areas

### 1. Orchestrator Singleton Reset in Tests (Stories 2.3, 2.4-INT)

**Impact:** High — 17 of 32 cross-story integration tests fail due to orchestrator singleton state management issues. The `_orchestrator` global singleton in `factory.py` doesn't reset cleanly between test runs. Each test must call `factory_module._orchestrator = None` in teardown, but some tests still interfere.

**Root Cause:** Python module-level singletons with mutable state are inherently difficult to isolate in parallel test runs. The factory pattern caches the orchestrator instance, and resetting it requires reaching into module internals.

**Action:** Refactor `get_tts_orchestrator()` to accept a dependency injection parameter or use a context-local factory. Add `reset_orchestrator()` function as a public test-only API. Priority: High for Epic 3.

### 2. In-Memory Session State Scalability (Story 2.3)

**Impact:** Medium — TTS session state is stored in a Python dict keyed by `vapi_call_id`. This works for single-instance deployment but will not scale horizontally. If the API runs behind a load balancer with multiple instances, session state is lost on failover.

**Root Cause:** Architectural simplicity — in-memory storage was the fastest path to deliver the MVP. Redis-based session state was discussed but deferred.

**Action:** Before horizontal scaling, migrate `_session_state` to Redis. The `reset_session()` and `get_or_create_session()` methods already have clean interfaces, so the migration should be mechanical. Track as technical debt for Epic 5 (Session Scaling).

### 3. Test Quality Inconsistency Across Stories

**Impact:** Medium — Test quality scores varied significantly:
- Story 2.3: 97/100 (A+) — the benchmark
- Story 2.4-INT: 92.6/100 (A) after remediation (was 86.4)
- Story 2.6: 86/100 (B) after remediation (was 67.75/100 D+)

Story 2.6 had the most severe initial quality score (D+) due to hard waits, broken test methods, and missing E2E coverage for AC6. While all stories reached production-ready quality after remediation, the initial variance suggests the review process catches issues that should be prevented upstream.

**Root Cause:** Different development sessions/agents produced tests with different quality baselines. No pre-generation quality template was enforced.

**Action:** Create a test quality template/checklist that must be followed before marking tests as complete. Include: no hard waits, factory functions, cleanup hooks, BDD naming. Run test quality review as part of the story completion workflow, not as a separate step.

### 4. Cross-Story Integration Testing Gaps (Story 2.4-INT)

**Impact:** Medium — While each story had strong unit test coverage, the seams between Stories 2.1, 2.2, and 2.3 were only tested after all three were "done." This meant integration issues (e.g., call-end ordering, TTS session reset before status update) were discovered late.

**Root Cause:** Integration tests were treated as a separate story rather than being written incrementally as stories were completed.

**Action:** For Epic 3, write integration tests concurrently with feature stories. When Story 3.2 (RAG Namespacing) is completed, immediately write tests for the 3.1→3.2 integration surface. Don't wait for a separate "integration test story."

### 5. Story 2.6 Documentation Volume (1,258 Lines)

**Impact:** Low — Story 2.6's implementation artifact grew to 1,258 lines, making it difficult to navigate. The file includes the original story, pivot documentation, multiple ADRs, test quality reports, and the change log. This violates the < 300-line guideline applied to test files.

**Root Cause:** The story underwent a pivot (calibration → presets) which doubled the documentation. Multiple quality review rounds added sections rather than replacing them.

**Action:** Use the `shard-document` skill to split large story artifacts into core story + appendices. Story files should contain the acceptance criteria, tasks, and dev record. ADRs, research, and test reports should be separate linked files.

## Key Insights

1. **Provider abstraction is the most valuable architectural pattern in Epic 2.** The TTSOrchestrator's strict interface meant that adding a third TTS provider (future) would require only implementing the abstract base class — zero changes to the orchestrator, webhook handlers, or session management. This pattern should be replicated for Epic 3's LLM provider abstraction.

2. **The pivot on Story 2.6 saved weeks.** Competitive analysis revealed that the industry standard (presets) was simpler and better validated than the original design (calibration). The team should build competitive analysis into the story creation workflow for all user-facing features.

3. **Fire-and-forget telemetry is a valid architectural choice when the tradeoff is explicit.** Story 2.4's decision to drop events rather than block the voice pipeline was controversial but correct. The key was making the tradeoff visible: degradation metrics (AC 8.5) ensure data loss is detectable and alertable.

4. **CSS-only animations are underutilized.** The Pulse-Maker component achieved 60fps animation with zero JavaScript overhead. This pattern should be the default for all visual feedback components. JavaScript animation should require justification.

5. **Test quality reviews catch real issues but should shift left.** Every story that underwent test quality review improved significantly (6–18 points). But the reviews consistently found the same issues: hard waits, missing factories, no cleanup hooks. These are preventable with a pre-generation checklist.

## Significant Discoveries

### 1. WebSocket Event Architecture Needs Extension (Story 2.5)

The `useTranscriptStream` hook was originally designed for transcript text only. Story 2.5 discovered it needed to handle `speech_state` WebSocket messages (speech-start, speech-end, interruption) as well. The hook was extended to create synthetic `TranscriptEntry` objects with `event_type` fields. This pattern will likely need further extension in Epic 3 for RAG citation events and in Epic 5 for sentiment analysis events.

**Recommendation:** Extract a generic `useWebSocketEvents` hook that can be composed with domain-specific parsers (transcript, speech, citation, sentiment).

### 2. Redis Is Now a Runtime Dependency (Story 2.6)

Story 2.6 introduced Redis for preset sample caching. Previously, the system had no Redis dependency. The `PresetSampleService` handles Redis unavailability gracefully (falls through to generation), but Redis is now in the critical path for performance.

**Recommendation:** Add Redis to the infrastructure requirements documentation. Include Redis health in the `/health` endpoint. Consider Redis for session state (replacing in-memory dict) as discussed in Challenge #2.

### 3. Binary Volume State Was Sufficient for MVP (Story 2.5)

The original spec for Pulse-Maker called for continuous volume interpolation from Vapi amplitude data. The team discovered that binary state (speaking: 0.8, idle: 0.0) with exponential decay produced a convincing visual effect. This suggests that MVP simplification doesn't always require compromise — sometimes the simpler approach is indistinguishable from the complex one.
