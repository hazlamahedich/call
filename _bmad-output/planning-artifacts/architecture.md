## Step 2: Project Context Analysis

Based on a collaborative "Party Mode" deep-dive with the architecture team, we have identified the following foundational constraints and considerations:

### Core Architectural Pillars
1. **The Latency Budget (<500ms):**
   - **Stream-First Design:** Direct gRPC/WebSocket pipelines between consumers, LLM providers, and telephony carriers.
   - **No-Buffer Middleware:** Avoidance of traditional queuing for active voice streams.
2. **Multi-layer Tenancy & Isolation:**
   - **Agency-First Hierarchy:** Support for Platform > Agency > Client Sub-account structures.
   - **Data Privacy:** Proposed use of PostgreSQL Row-Level Security (RLS) as the baseline, with potential schema isolation for high-burst enterprise agencies.
3. **Regulatory Safety (TCPA/DNC):**
   - **Real-time Guardrails:** Implementation of a "Call-Router Guardrail" that validates consent and time-windows before SIP initiation.
4. **Intelligent Onboarding:**
   - **RAG Inheritance:** A "Template-to-Instance" model for knowledge bases to meet the 10-minute setup promise.

## Step 3: Starter Template Evaluation

### Primary Technology Domain
**Full-stack AI Voice SaaS** based on real-time latency and multi-tenant agency requirements.

### Starter Options Considered
- **VapiBlocks Next.js Starter:** Excellent for rapid frontend calling state but lacks enterprise backend type-safety and compliance isolation.

## Step 5: Implementation Patterns & Consistency Rules

To ensure consistency across multiple AI agents and human contributors, the following patterns are mandated:

### 1. Naming & Type Safety (The "Bridge")
*   **Database/Backend:** `snake_case` (e.g., `agency_id`).
*   **JSON/Frontend:** `camelCase` (e.g., `agencyId`).
*   **Serialization Rule:** Python models use `AliasGenerator` to automatically convert `snake_case` to `camelCase` in outgoing JSON.
*   **Sync Logic:** `packages/types` contains autogenerated TypeScript interfaces from `SQLModel` definitions. agents **MUST** run `turbo run types:sync` after schema changes.

### 2. Error Handling & Constants
*   **Standardized Errors:** `{ "error": { "code": "STRING_ID", "message": "Human readable" } }`.
*   **Shared Registry:** `packages/constants` holds a `codes.json` mapping that both `apps/api` and `apps/web` use to prevent string-mismatch bugs.

### 3. Project Structure (Monorepo)
*   `apps/web`: Next.js 15 (Dashboard/Cockpit).
*   `apps/api`: FastAPI (Voice/Logic).
*   `packages/compliance`: Shared logic for DNC/TCPA validation. **Mandatory import** for all call-initiation routes.
*   `packages/types`: Shared TypeScript interfaces.
*   `packages/constants`: Shared error codes and config keys.

## Step 6: Advanced Verification & Performance Guardrails

Based on the technical deep dive (Murat, Winston, Saga), the following performance and quality protocols are integrated:

### 1. Latency Gates (CI/CD)
*   **TTFB Gate:** A dedicated GitHub Action must benchmark `packages/voice-engine` processing. Build fails if internal processing TTFB > 100ms.
*   **Load Simulation:** Use **Neon Branching** to spin up ephemeral performance test environments with 50,000 lead records to verify RLS performance under load.

### 2. Edge-Streaming Architecture
*   **Compliance Middleware:** DNC and TCPA status must be cached in-memory (Redis) at the `Streaming Gateway` level to avoid database round-trips during an active call.
*   **Observability:** Use asynchronous telemetry collectors ("Sidecars") for call metadata to ensure the main voice event loop is non-blocking.

### 3. The "Aha!" Moment Optimizations
*   **First 15 Prefetching:** The RAG pipeline must generate the initial personalization context *before* the outbound SIP initiation. 
*   **Graceful Degeneracy:** Implementation of pre-cached 'Edge Fillers' (thinking sounds/filler phrases) to manage up-to-300ms retrieval spikes without silence.

### 4. Tenancy & Sync
*   **Isolation Mapping:** `Clerk Organizations` -> `Agency`. `PostgreSQL RLS` enforced on every query via mandatory `tenant_id` context.
*   **Sync Logic:** Autogenerated TypeScript interfaces from `SQLModel` via `turbo run types:sync`.

## Step 7: The "Master Join" Zero-Latency Protocol

To achieve the <200ms "Master Join" requirement, the architecture implements a "Pre-emptive Execution" model rather than a "Request-Response" model.

### 1. Shadow-Bridging & Dual-Zone Hysteresis
- **Predictive Audio Buffer (PAB):** The Voice Engine initiates an audio bridge (SIP) silently at **60% confidence** (Zone 1). This "Pre-warm" happens in the background.
- **Human Alert Trigger:** The agent is only notified/alerted via the UI/Headset at **85% confidence** (Zone 2).
- **Perceived Latency:** By the time the human clicks "Join," the audio bridge is already established, reducing perceived join time to **<50ms**.

### 2. Metadata Bursting (gRPC via QUIC)
- **Protocol:** Use **QUIC** transport to avoid head-of-line blocking. 
- **The Burst:** The "Context Triad" (Why, Mood, Objective) is delivered on a high-priority gRPC stream ID **<30ms** before audio completion.
- **UI Interaction:** This ensures the "Atmospheric Magic" feedback appearing in the cockpit is ahead of the audio connection.

### 3. False Positive Safety (The "Ghost" Guardrails)
- **Muted Entrance:** All human joins start with a **2-second Muted Vibe Check**. The agent hears the call but their mic is dead until they confirm or the timer expires.
- **Tactical Abort:** Anomaly detection monitors for immediate lead sentiment reversals post-join to bridge-kill before human interaction.
- **Return to Shadow (RTS):** A non-destructive exit that preserves AI continuity.

### 4. RAG Context Stitching
- **Synthetic Catch-up:** Upon an RTS event, the LLM is re-prompted with a summary of the human's input to ensure the AI's next response follows the human's logic.
- **Role Tagging:** Transcript segments are strictly tagged as `assistant-human` or `assistant-ai` for future retrieval accuracy.

## Step 8: External Data Enrichment Architecture (FR8)

To fulfill the dynamic icebreaker and lead enrichment requirement, the system integrates a dedicated EnrichmentEngine.

### 1. Provider Selection
- **Contact Enrichment:** **Apollo.io API** for real-time lead verification and attribute retrieval (Job title, Company size, Recently raised funds).
- **Real-time Icebreakers:** **Perplexity API (Sonar-Huge)** for searching current news, LinkedIn posts, and company press releases within the last 24 hours.

### 2. The Enrichment Engine Pipeline
- **Trigger:** Initiated during the "Pre-flight Calibration" phase of onboarding or 30s before a scheduled outbound burst.
- **Processing:**
    1. `LeadDataFetcher` queries Apollo for contact metadata.
    2. `NewsAggregator` search queries Perplexity for recent company/person relevance.
    3. `ContextSynthesizer` passes the combined data to the RAG pipeline to generate 3 "Tactical Icebreakers."
- **Storage:** Results are cached in **Redis** with a 12-hour TTL to ensure freshness without redundant API costs.

## Step 9: Telephony Failover Protocol (NFR.R2)

To meet the <3s session-persistent failover requirement, the architecture implements dual-channel signaling and state mirroring.

### 1. Hybrid Vapi-Twilio Signaling
- **Primary:** **Vapi Voice Engine** (Low-latency WebRTC/SIP bridge).
- **Recovery Fallback:** **Twilio SIP Interface** configured as a secondary trunk.

### 2. State Mirroring (The "Shadow State")
- **Redis Session Store:** Every active call session mirrors its "Conversation State" (LLM context, last transcript, agent role) to a Redis global cluster.
- **Heartbeat Monitor:** A Sidecar service monitors Vapi connection health. If a packet loss threshold (>15% over 500ms) is detected:
    1. **Re-route:** The Backend issues a `SIP REFER` or `RE-INVITE` to the fallback carrier.
    2. **Context Inject:** The fallback voice engine retrieves the "Shadow State" from Redis and resumes the LLM stream within <2s.
- **User Experience:** The Human Agent remains in the Obsidian Cockpit; the UI displays a subtle "Stabilizing Link" overlay during the handover.
